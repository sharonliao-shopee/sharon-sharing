## Page 3 | Quick Start for New

### Slide title

Start with one real, small task and build your own way of working with Codex

### Slide description

You do not need to learn every feature at once. Connect the systems you use most, complete one useful deliverable, and keep the methods that work.

### Step 1 | Install the essential plugins

Start with the documents, email, and calendar tools you use every day, so Codex can access real work materials within your permissions.

### Step 2 | Start with a simple task

Choose something already on your list this week, with real source material and an outcome that is easy to judge—for example, a brief, option comparison, or meeting action list.

### Step 3 | Give clear instructions

Explain the context, objective, audience, sources, requirements, and boundaries. Work in rounds: confirm the brief, review a first draft, follow up, then finalize.

### Step 4 | Start codifying your method

Turn the criteria and formats you repeat into a template, then gradually upgrade it into a reusable Skill.

### A 30-minute minimum viable first run

Connect Google Drive → Ask Codex to find recent analysis on warehouse capacity and summarize it in a one-page brief → Ask, “What are the key insights?” → Add analysis based on the response → Generate an HTML report.

### Slide takeaway

Plugins define what Codex can access. Instructions define what to do now. Skills define how the work should be done next time.

## Page 3-2 | Start with a Simple Task

### Slide title

Start with a simple task: Choose work you can use immediately

### Slide description

For a first experience, complexity matters less than real source material, a clear outcome, and your ability to judge whether the result helps.

### Task A | One-page project brief

#### Best for

Quickly understand the project context, progress, risks, and decisions needed.

#### Sample prompt

> Review @Project Materials and summarize:  
> 1. Problem to solve  
> 2. Current progress and decisions  
> 3. Risks, dependencies, and gaps  
> 4. Critical actions for the next two weeks  
> 5. Decisions I need to make  
> Every number and conclusion must trace back to a source.

### Task B | Option comparison

#### Best for

Compare options on a common basis and make the assumptions behind the recommendation explicit.

#### Sample prompt

> Review @Project A and @Project B. First define a common comparison framework, then compare:  
> - Benefits and costs  
> - Implementation difficulty  
> - Risks and prerequisites  
> Close with your recommendation, assumptions, information gaps, and what the pilot should validate.

### Task C | Meeting action list

#### Best for

Turn meeting notes into decisions, actions, PICs, and due dates.

#### Sample prompt

> Review @Meeting Transcript and extract:  
> - Confirmed decisions  
> - Unresolved disagreements  
> - Action / PIC / Due date  
> - Missing data or materials  
> - Items due before the next meeting  
> If no PIC or date is explicit, mark it “To confirm.”

### Slide takeaway

Choose something already due this week + real source material + an easy-to-judge result + an output you can use immediately.

## Page 3-3 | Give Clear Instructions

### Slide title

Give clear instructions: Define the context, delivery standard, and boundaries

### Slide description

Brief Codex as you would a new colleague: the clearer the context, the more usable the result.

### Six parts of a clear instruction

1. Context: Why this is happening
2. Objective: What needs to be solved
3. Audience: Who will read and decide
4. Sources: What Codex should review
5. Requirements: Length, structure, and format
6. Boundaries: What Codex must not assume and which actions need confirmation

### Full example | Executive project update

> **Context:** We want to propose a new project to leadership, but the supporting materials are scattered.  
> **Objective:** Help leaders understand in 10 minutes why we should do it, what the proposal is, and what support we need.  
> **Sources:** Review the project brief, research notes, and available data.  
> **Requirements:** Start with a storyline of no more than five slides. Use a conclusion-led title on each slide. Build the full content only after I confirm the storyline.  
> **Boundaries:** Do not invent data. Flag anything that cannot be verified. Ask before editing the source file or sending anything externally.  
> When complete, review it from a leader’s perspective: Which claims lack evidence, and which questions are most likely to come up?

### Slide takeaway

Give clear instructions, but do not try to make the first prompt perfect.

## Page 3-4 | Start Codifying Your Method

### Slide title

Start codifying your method: From repeated instructions to a reusable Skill

### Slide description

After three to five similar tasks, capture the stable input requirements, steps, decision criteria, output format, and points that still require human confirmation.

### Path to reuse

Real tasks → Repeated preferences → Template → Skill

1. **Identify the real task:** Complete several similar tasks first.
2. **Extract repeated preferences:** Find the requirements you keep emphasizing.
3. **Create a template or SOP:** Fix the input requirements, steps, references, and output format.
4. **Create a Skill:** Package the method, then keep running and improving it through real use.

### Sample prompt | Turn experience into a template

> Review the similar tasks we completed recently and extract the points I repeatedly emphasized:  
> 1. Required inputs  
> 2. Analysis or execution steps  
> 3. Decision criteria  
> 4. Output format  
> 5. Boundaries that require human confirmation  
> First, turn them into a reusable template and flag anything that still needs my confirmation.

### Plugin vs. Skill

- Plugin: Which systems and source materials Codex can access
- Skill: Which method Codex should follow for this type of work

### Slide takeaway

Start small, use it on real work, and keep the methods that prove effective.
